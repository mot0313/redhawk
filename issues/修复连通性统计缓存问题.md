# 修复连通性统计缓存问题

## 问题描述
有时候获取连通性统计会直接返回上一次结果，再次点击不会进行检测。即使前端传递了`useCache=false`参数，后端仍然返回缓存的数据。

## 问题分析
通过日志分析和代码检查，发现问题的根本原因是**参数名称不匹配**：

- **前端传递的参数**：`useCache: false`, `cacheTtlMinutes: 5`（驼峰命名法）
- **后端接收的参数**：`use_cache: bool`, `cache_ttl_minutes: int`（下划线命名法）

由于FastAPI的Query参数默认不会自动转换驼峰到下划线，导致前端传递的参数没有被后端正确接收，总是使用默认值`use_cache=True`。

## 修复方案
为后端API参数添加别名，同时支持驼峰和下划线命名：

```python
# 修改前
use_cache: bool = Query(default=True, description="是否使用缓存")
cache_ttl_minutes: int = Query(default=5, ge=1, le=60, description="缓存时间（分钟）")

# 修改后
use_cache: bool = Query(default=True, alias="useCache", description="是否使用缓存")
cache_ttl_minutes: int = Query(default=5, alias="cacheTtlMinutes", ge=1, le=60, description="缓存时间（分钟）")
```

## 修复结果
1. **向后兼容**：既支持原有的下划线命名，也支持前端的驼峰命名
2. **参数正确传递**：前端的`useCache=false`现在能正确映射到后端的`use_cache=False`
3. **缓存机制正常**：当`useCache=false`时，系统会跳过缓存进行实时检测

## 测试验证
- 添加了日志记录，可以确认参数是否正确接收
- 前端点击"获取连通性统计"时，如果`useCache=false`，会执行实时检测
- 缓存机制仍然正常工作，提供性能优化

## 涉及文件
- `backend/module_redfish/controller/connectivity_controller.py`：添加参数别名
- `frontend/src/api/redfish/connectivity.js`：前端API调用（无需修改）
- `frontend/src/views/redfish/device/index.vue`：前端调用逻辑（无需修改）

## 注意事项
- 此修复保持了完全的向后兼容性
- 建议在其他API中也考虑统一的参数命名规范
- 可以考虑使用全局的参数转换中间件来处理类似问题
